# Whiteboard

대화가 복잡해진 순간 — 디버깅하다, 설계하다, 선택지를 저울질하다 글이 길어져 따라가기 힘들 때 — 지금 논의를 **한 장의 자기완결 HTML**로 외부화한다. 화이트보드에 같이 그려가며 정리하듯, **무엇을 그릴지 대화에서 먼저 합의**하고 그 위에 시각물을 올린다.

이 스킬의 본질은 렌더링이 아니라 **대화 독해**다. 시각화가 실패하는 가장 흔한 원인은 못생겨서가 아니라 *틀린 것을 그려서*다. 그러니 화려한 산출보다 "무엇을 그릴지"가 먼저다.

## 이 스킬이 하는 일 / 안 하는 일 (책임 경계)

- **한다**: 대화에서 막힌 지점을 읽어 *무엇을 어느 수준으로* 그릴지 합의 → 내용 유형에 맞는 매체(비교 매트릭스·다이어그램·차트·타임라인·KPI)로 라우팅 → 커밋된 미학의 자기완결 단일 HTML 1장 생성 → 실제 렌더 검증.
- **안 한다 — 사용자 대신 결정**: 선택지를 *비교해 보여주되* 무엇을 고를지는 정하지 않는다. 추천과 근거는 시각물에 담되, 결정은 사용자 몫이다.
- **안 한다 — 질문 바꿔치기**: 사용자가 "설명해줘 / 이해하게 해줘"라고 하면 그건 *이해*를 원하는 것이다. 이걸 "Option A vs B 결정 자료"로 점프하지 않는다. 사용자가 물은 층위로 답한다.
- **안 한다 — 코드 어휘로 그리기**: 사용자가 제품·도메인 언어로 생각하는데 `isNull`·`dead code 8곳`·스키마 이름으로 그리면 더 헷갈린다. 사용자가 쓴 단어로 그린다.
- **안 한다 — 추측을 사실로**: 코드만 보고 "이 뱃지는 뜬다" 같은 단정을 그리지 않는다. 시각물에 들어가는 사실은 확인된 것만 — 미확인은 "미확인"으로 표시하거나 사용자에게 묻는다.

> 이 경계들은 추상적 원칙이 아니라 실제 실패에서 도출됐다: 사용자가 *개념*("이게 뭔지")을 물었는데 *옵션 결정 매트릭스*를 들이밀고, 코드 어휘로 설명하고, 확인 안 된 UI 상태를 단정해, 멀쩡히 렌더된 HTML 두 장을 보고도 사용자가 끝내 이해하지 못한 사례가 이 스킬의 출발점이다. 산출물을 아무리 다시 만들어도 소용없었다 — **대화 흐름에서 무엇을 그릴지가 틀렸기 때문**이다.

## 데이터 ≠ 로직 경계

이 스킬 본문은 **로직**(어떻게 읽고·무엇을 그릴지 정하고·어떻게 렌더하는가)만 담는다. 생성한 HTML 산출물은 사용자가 지정한 경로(없으면 현재 작업 디렉토리, 또는 일회성이면 `/tmp`)에만 쓰고, 이 스킬이 설치된 레포에는 절대 쓰지 않는다. 산출물에 사용자의 코드·도메인 내용이 들어가므로 더욱 그렇다.

## 워크플로우

### 0. 게이트 — 그리기 *전에* 대화에서 합의 ★ 이 스킬의 핵심

요청을 받자마자 HTML을 만들지 않는다. 먼저 세 가지를 대화에서 확정한다. 이게 통과돼야 다음 단계로 간다.

**(a) 무엇이 막혔나 — 1문장으로 되받아 확정.**
지금 사용자가 풀려는 게 다음 중 무엇인지 짚는다. 유형이 곧 매체를 정한다:

| 막힌 유형 | 사용자 신호 | → 매체 |
|---|---|---|
| ① 개념 — "이게 뭔지" | "○○가 뭐야", "△랑 □ 차이가 뭔데" | 도식·비유·2축 매트릭스(단, 사용자 언어로) |
| ② 결정 — "뭘 고를지" | "뭘로 가지", "어느 쪽이 나아" | 비교 매트릭스 + 추천 핀 |
| ③ 흐름 — "순서·인과" | "왜 가끔 터지지", "어떻게 도나" | 시퀀스/플로우 다이어그램 |
| ④ 비교 — "어떻게 다른지" | "이것들 차이가" | 비교표·매트릭스 |
| ⑤ 수치 — "얼마나/어떻게 움직이나" | 데이터·추세·랭킹 | 차트 |

대화 맥락에서 유형이 분명하면 바로 정하고, **모호하면 추측하지 말고 1개의 분기 질문으로 확정한다.** "지금 헷갈리시는 게 ① ○○가 *뭔지*인가요, ② 어느 쪽으로 *정할지*인가요?" 한 번 묻는 게, 틀린 걸 그려 여러 턴 왕복하는 것보다 항상 싸다.

**(b) 추상화 레벨을 사용자 언어로 맞춘다.**
시각물에 쓸 어휘를 사용자가 대화에서 쓴 단어로 잡는다. 코드 내부 용어(타입명·함수명·"dead code N곳")가 아니라 제품·도메인 언어로. "이걸 사용자의 말로 하면 무엇인가"가 핵심 질문이다.

**(c) 핵심 질문 1개로 좁힌다.**
"이 한 장이 *무슨 질문에 답하면* 되는가"를 한 문장으로 정한다. 이게 시각물의 제목이자 상단 고정 요소가 된다. 답할 질문이 둘 이상이면 한 장에 욱여넣지 말고 무엇이 우선인지 정한다.

### 1. 매체 계획 — 섹션별로, 생성 전에

만들 한 장을 섹션으로 쪼개고 **각 섹션에 비-prose 매체를 배정**한다(위 표의 매체). prose(산문)는 매체로 표현 못 하는 짧은 맥락에만, 정당화될 때만. "모든 게 카드에 담긴 글줄"이 되는 기본 실패를 막는 게 이 계획 단계다. 핵심 질문에 대한 답·추천은 **맨 위에 고정**한다 — 사용자가 스크롤 끝까지 가야 결론을 만나게 하지 않는다.

### 2. 사실 우선 — 데이터/상태를 먼저 확정

차트·매트릭스에 들어갈 값은 먼저 JSON/표로 적고 그 위에 렌더한다. UI 상태·동작처럼 **확인이 필요한 사실은 추측하지 말고 확인**한다(코드를 읽거나 사용자에게 묻는다). 확인 못 한 건 "미확인"으로 둔다 — 그려 넣고 단정하지 않는다.

### 3. 미학 커밋 + anti-slop

`references/design-system.md`에서 내용 톤에 맞는 미학 패밀리 1개를 고르고, anti-slop 금지목록을 적용한다. 코딩 전에 방향을 정한다(매번 새 generic 레이아웃을 즉흥으로 만들지 않기). 진지한 맥락에 과한 스타일(brutalist 등)은 피하고, 사용자가 패밀리를 지정하면 그걸 따른다.

### 4. 스택 라우팅

`references/design-system.md`의 라우팅 표대로 매체별 최소 라이브러리를 쓴다(Tailwind·Chart.js·Mermaid·Markmap·Alpine.js, 전부 CDN 1태그·무빌드). D3 손코딩·Recharts(빌드 필요)는 피한다.

### 5. 단일 HTML 출력

인라인 CSS/JS, CDN 라이브러리, 반응형, 다크/라이트, 인쇄(PDF) 친화로 **자기완결 .html 한 장**을 쓴다. 데이터는 인라인 임베드(오프라인에서도 열림). 골격은 `references/design-system.md`의 템플릿 참고.

**Mermaid를 쓸 땐 반드시 견고하게**: 동적 import면 `startOnLoad:false`로 두고 import 완료 후 `await mermaid.run()`을 **명시 호출**한다. `startOnLoad:true` + 동적 import는 자동 렌더 타이밍을 놓쳐 다이어그램이 raw 소스로 떨어지는 간헐 실패를 낳는다(브라우저마다 됐다 안 됐다). 라벨의 특수문자(`✕ → — '' "" ()`)는 파싱을 깨뜨리니 일반 텍스트로 둔다.

### 6. 렌더 검증 — 눈으로 확인했다고 가정하지 않는다

`scripts/verify_render.sh <html>`로 실제 렌더를 검증한다. Mermaid가 raw로 남았거나 차트가 안 그려졌으면 FAIL과 사유가 나온다 — **고친 뒤 다시 검증**한다. "파일 만들었으니 됐겠지"로 넘기지 않는다. ("텍스트 벽을 깨자"는 산출물이 텍스트 벽이 되는 게 정확히 이 단계를 건너뛸 때 생긴다.) Chrome이 없으면 검증을 건너뛰고 사용자에게 브라우저로 직접 확인을 요청한다 — 검증 못 한 걸 "됐다"고 하지 않는다.

### 7. 전달

산출물 경로를 알리고, `open`(macOS) 등으로 띄운다. 핵심 질문에 답이 됐는지 한 줄로 짚고, 사용자가 보고 방향을 틀 수 있게 한다.

## 스크립트

- **`verify_render.sh <html>`** — 생성한 HTML을 headless Chrome으로 렌더해 Mermaid/Chart.js가 *실제로 그려졌는지* 검증한다(raw 잔존·미렌더 감지). 종료코드 0=통과, 1=렌더 실패(사유 출력), 3=Chrome 없음(검증 불가). headless가 통과해도 타이밍 레이스는 못 잡으니, 5단계의 견고한 코드(명시 `mermaid.run()`)가 1차 방어이고 이건 2차다.

## 원칙 (요약)

- **대화 독해 > 렌더링.** 무엇을 그릴지가 어떻게 그리는지보다 먼저다. 틀린 걸 예쁘게 그리면 더 나쁘다.
- **설명 요청을 결정 요청으로 바꾸지 않는다.** 사용자가 물은 층위로 답한다.
- **사용자 언어로 그린다.** 코드 어휘는 사용자를 막는다.
- **추측을 그리지 않는다.** 시각물의 사실은 확인된 것만.
- **렌더를 검증한다.** 만들었다고 됐다고 하지 않는다.
- **가독성 하한.** 어떤 텍스트도 13px 미만으로 두지 않는다(테이블 헤더·캡션·라벨이 가장 자주 위반). 다이어그램은 컨테이너 가운데 정렬. 디테일은 `references/design-system.md`.
- **단순한 건 단순하게.** 3줄로 끝날 답을 대시보드로 만들지 않는다 — 한 장이 과하면 짧은 단일 매체(다이어그램 하나, 표 하나)로 충분하다.
