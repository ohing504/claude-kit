# ai-doc-improver

AI agent가 읽는 문서를 두 축 — **토큰 밀도**(효율적으로 읽히나)와 **instruction 준수율**(지시를 잘 따르나) — 으로 다듬는다. 대상은 CLAUDE.md·SKILL.md·에이전트 정의·README·spec·코드 주석 등 *AI가 읽는 문서*. 사람이 읽는 산문·마케팅 글은 대상이 아니다.

## 흐름

audit → 개선안 제시 → 확인 → 리라이트. 확인 없이 리라이트하지 않는다.

1. **대상 확정** — `$ARGUMENTS`가 있으면 `Glob($ARGUMENTS)`로 파일 존재 확인. 파일이 존재하면 그 경로를 대상으로 바로 진행; 존재하지 않거나 `$ARGUMENTS`가 없으면 사용자에게 대상 확인. 여러 파일이면 파일 단위로 진행.
2. **audit** — 대상 타입이 *타입별 특화* 가이드를 가지면 해당 `references/` 파일을 Read하고 범용 점검 신호에 더해 적용한다.
3. **개선안 제시** — 후보를 *항목별로*. 각 항목은 대상 본문을 그 자리 인용 + 개선안 + 추천. 한 항목만 보고 결정 가능하게.
4. **확인** — 사용자가 적용할 것을 고른다. 다이어트는 삭제가 우세하므로 통째 적용보다 항목별 확인이 안전.
5. **리라이트** — 합의된 것만 반영.

## 점검 신호

### 토큰 밀도 (효율)

- 산문에 다층 정보·절차가 접속사·콤마로 뭉침 → **구조화**: 표제+sub-bullet 계층, 규칙·매핑(`A→B`), 단언 (군더더기 연결어 제거)
- 같은 내용이 두 곳 이상 중복 → 한 곳만 두고 나머지는 **pointer**(파일·앵커 인용)
- 배경 설명·메타 서술·자기 자랑·다른 도구 정의 인용 → **삭제**
- 본문이 비대 → 상세를 `references/`로 **외부화**하고 본문엔 pointer. **단, 스킬이 매 실행마다 참조하는 내용은 인라인 유지** — 분리하면 실행마다 Read 비용이 추가된다.
- 코드·식별자·시그니처가 이미 말하는 내용을 문서가 반복 → **삭제** (자명한 것을 서술하지 않는다)
- 재발하지 않는 일회성 이력(특정 커밋 SHA·해결된 버그 서술·죽은 TODO) → **삭제** (현재 시점 SSOT만 남기고 이력은 changelog·git 영역)
- 한 단계·섹션이 여러 독립 관심사를 평면 나열 → 관심사·시점 축으로 **하위 그룹화** (라벨이 의미를 갖고 개별 규칙이 본문 중간에 묻히지 않게 — lost-in-the-middle)

### instruction 준수율 (프레이밍) — [`references/instruction-adherence.md`](references/instruction-adherence.md)

- 부정문·금지가 길게 서술됨 → **긍정형 지시로 뒤집고**, 남는 금지는 한 줄로. "하지 마라"보다 "이렇게 하라"가 더 안정적으로 따라진다.
- 금지·예외가 본문에 묻힘 → **분리해 명시하되 짧게**. 장황한 금지 단락은 그 표현을 활성화(salience)해 역효과 — 가시성(분리)과 짧음(salience 억제)을 동시에.
- 추상 규칙·경계 모호, 또는 톤·출력형·판단이 서술로 안 잡힘 → **good/bad 대조 예시**로 시연. 스킬이 짧은 쌍을 후보로 초안 → 항목별 확인 후 반영. 게이트: 고애매 AND 반복 오해 한정(자명한 룰엔 생략). 효과는 이모지가 아니라 *구조적 대조*(`Good:`/`Bad:` 라벨도 동일).

## 타입별 특화

대상이 특정 타입이면 범용 점검에 더해 `references/<타입>.md` 특화 가이드를 적용한다.

- **CLAUDE.md** (및 변형 `.local.md`, `~/.claude/CLAUDE.md`) → [`references/claude-md.md`](references/claude-md.md)
- **SKILL.md** → [`references/skill-md.md`](references/skill-md.md)

특화 가이드는 그 타입 고유의 삭제 신호·형식 하드 검사(정규식·길이·키 등 기계 판정)·**핵심 누락 플래그**를 정의한다. 누락은 *플래그만* 한다 — 무엇을 채울지 추가는 사용자 확인이고, 이 스킬은 군더더기 제거가 주력이라 내용을 창작하지 않는다. 유일한 예외는 준수율용 good/bad 대조 예시 초안 — 항목별 확인 후 반영한다.

## 원칙

- **측정 단위는 토큰 밀도, 줄 수는 중립** — 줄 수 감소가 목표가 아니다. 분해로 줄 수가 늘어도 연결어가 빠지면 토큰은 준다.
- **한 항목 = 한 의미 단위** — 의미로 분해한다. 한 룰을 억지로 여러 줄로 쪼개 맥락을 끊지 않는다.
- **핵심을 앞에** — 긴 문서의 중간 정보는 묻힌다(lost-in-the-middle). 표제에 핵심·grep 앵커를 두고 조건·예외는 sub-bullet로.
- **deletion 우세** — 다이어트는 재배열이 아니라 삭제·외부화. 개선의 실질은 줄어든 분량이다.
- **프레이밍 우선순위** — 긍정형 지시 > (경계 모호·시연이 나을 때) 짧은 대조 예시 > 짧은 명시적 금지. 부정 제약을 길게 쓰지 않는다(negation salience). 단 frontier 모델은 금지도 잘 따르므로 긍정형은 "더 안정적"인 정도지 절대 규칙은 아니다.

## 하지 않는 것

- 줄 수를 줄이려 여러 줄을 한 줄로 이어붙이기 — prose화라 토큰·노이즈가 오히려 는다. 안티패턴.
- 의미·내용 손실 — 형식 결함이면 형식만 손댄다. 내용을 지우려면 사용자 확인.
- 확인 없는 대량 삭제·일괄 리라이트.
- 사람이 읽는 산문 문서에 적용 — 이 스킬은 AI 대상 문서 전용.

## Edge Cases

- **이미 잘 쓰인 문서**: 점검 신호가 하나도 안 걸리면 그대로 통과 보고 — 억지로 건드리지 않는다.
- **삭제 후보가 다른 문서와 교차 링크**: 링크 끊김이 생길 수 있으니 삭제 전 역참조 확인(`grep`). 보고할 때 연결 영향도 함께 안내.
- **외부 관리 마커 (START/END)**: `CLAUDE.md` 등에 자동 동기 구간이 있으면 그 안은 편집하지 않는다. 마커 밖 영역에만 적용.
- **여러 파일 묶음**: 한 번에 다 고치지 않는다 — 파일 단위로 진행하고 각각 사용자 확인 받기.
