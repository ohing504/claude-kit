# memory-routing SSOT

memory-manager 스킬이 참조하는 메모리 분류·비용·audit·revise 규칙의 단일 정본. SKILL.md `## 적용하는 규칙`·`## 동작`이 이 문서를 SSOT로 참조한다.

## 1. 메모리 작동 모델 (전제)

- 메모리 경로: `~/.claude/projects/<cwd-encoded>/memory/` — 프로젝트(cwd)별 격리.
- 인코딩: cwd의 `/`·`.`를 `-`로 치환. `.`도 치환되므로 `~/.claude` 류 경로는 `--`가 생긴다(예: `/Users/<you>/.claude` → `-Users-<you>--claude`). 도출 `pwd | sed 's#[/.]#-#g'`.
- 매 세션 자동 로드: 그 프로젝트의 `MEMORY.md` 인덱스 *한 개만*. 개별 메모리 파일은 자동 로드 아님 — 관련 시 recall로 `<system-reminder>`에 surface.
- 전역 항상-로드 레이어: `~/.claude/CLAUDE.md` (모든 프로젝트 모든 세션 로드, 수기 관리).
- 저장 주체: 메모리는 Claude가 세션 중 *자율적으로* 저장한다(사용자 동의·지시 없이). 저장 시점에 보편/고유를 가르는 장치는 없다.
- 함의: 비용은 *프로젝트당 인덱스 길이*가 좌우(누적 폭발 아님). 낭비의 핵심은 *보편 규칙의 cross-silo 중복*이며, 자율 저장 탓에 이 중복은 구조적으로 쌓인다 — 그래서 본 시스템은 사후 정리(audit/revise) 도구다.

## 2. 3-tier 분류 기준

저장 직전에 사용자에게 묻는 라우팅 서비스가 아니다 — audit이 *오배치*를 판정하고 revise가 *통합 목적지*를 정할 때 쓰는 내부 잣대다.

| tier | 위치 | 들어갈 것 | 성격 | 로드 비용 |
|---|---|---|---|---|
| 1 | `~/.claude/CLAUDE.md` | 모든 프로젝트 공통 "일하는 방식" | feedback (*어떻게 일해라*) | 모든 세션 (비쌈 — 절제) |
| 2 | 프로젝트 `memory/` | 그 레포 고유 사실·결정·함정 | fact / decision (*이 레포는 이렇다*) | 그 프로젝트만 |
| 3 | (저장 안 함) | 코드·git·CLAUDE.md가 이미 담은 것, 일회성 | — | 0 |

**판정 질문**: "이 규칙이 다른 프로젝트에서도 똑같이 필요한가?"
- 예 → tier 1 (단, 이미 CLAUDE.md에 있으면 통합 불필요)
- 아니오, 이 레포 고유 → tier 2
- 코드·git·문서로 재도출 가능 → tier 3

**잣대는 보편성, 종류 아님**: 네이티브 메모리 종류(user/feedback/project/reference)가 tier를 직접 가르지 않는다. 같은 보편성 질문을 적용한다 — feedback 성격(커밋·푸시·어휘·세션 종료·리뷰 습관·문서 톤·검증 절차)은 거의 tier 1, fact/decision 성격(가격 tier·사업자 정보·배포 명령·아키텍처 결정·특정 버그 회귀·인프라 자격증명 위치)은 거의 tier 2로 *수렴*하지만, 최종 판정은 보편성이다.

**tier 1 절제 룰**: tier 1이라고 전부 CLAUDE.md로 통합하지 않는다. 전역은 *모든 세션* 비용이므로 *반복 빈도 높은 진짜 보편*만. 일회성·맥락 의존 feedback은 통합 대상 아님.

## 3. 저장 형식 (tier 2 프로젝트 메모리)

revise가 재배치·통합 시 네이티브 메모리 형식을 보존하기 위한 참조.

- 메모리 파일: frontmatter(`name`/`description`/`metadata.type`) + 본문 1개 사실. feedback/project는 본문에 `**Why:**`·`**How to apply:**` 동반.
- `MEMORY.md` 인덱스: 한 줄/항목(`- [제목](file.md) — 훅`). 콘텐츠 본문 금지. lean 유지.
- 관련 메모리는 본문에서 `[[name]]`로 링크.

## 4. audit 점검 항목

| 항목 | 탐지 방법 | 처리 |
|---|---|---|
| cross-silo 중복 | 파일명 키워드(commit/vocab/push/session 등) cross-silo 집계 + 인덱스 본문 취지 대조 | tier 1 통합 + 중복 silo 삭제 |
| 오배치 | 프로젝트 silo의 feedback 중 보편 규칙 식별, CLAUDE.md 기존 규칙과 대조. 홈 `.claude` silo는 우선 점검 | 전역 통합 또는 재배치 |
| 인덱스 bloat | `MEMORY.md` 줄 수 또는 바이트 어느 쪽이든 초과(대략 25줄 또는 3KB), 또는 줄당 평균 100자 초과(본문 혼입) | 머지·삭제·본문 분리 |
| 재저장 중복 | CLAUDE.md 섹션 헤딩별 같은 취지 대조 + 메모리가 언급한 식별자·키워드를 `grep`으로 코드·git에서 확인 | 삭제 |
| stale | 메모리가 언급한 파일·식별자·플래그의 현존 여부(`grep`·`ls`) | 삭제 또는 갱신 보고 |
| 인덱스 일치 | `MEMORY.md` 항목 ↔ 실제 `*.md` 파일 diff | 고아 라인 제거 / 미인덱스 파일 처리 |

**중복 판단 주의**: 파일명만으로 단정 금지. `commit_cadence`·`no_auto_commit`처럼 이름이 달라도 같은 행동 규칙이면 중복. 반대로 이름이 비슷해도 레포별 뉘앙스(브랜치 전략 차이 등)가 다르면 보존. 본문 취지 대조 필수.

## 5. revise 안전 규칙

- 삭제·통합·이동 전 *대상 목록*을 사용자에게 보고하고 확인. overwrite 금지, 삭제는 명시 결정 시만.
- 전역 통합은 CLAUDE.md의 *기존 섹션 구조를 런타임에 읽어* 규칙 성격에 맞는 섹션에 간결히 1회 — 특정 섹션 이름을 가정하지 않고 실제 헤딩을 보고 매칭, 적합 섹션 부재 시 사용자에게 어느 섹션(또는 신설)인지 확인. 외부 도구가 자동 동기하는 관리 구간(START/END 마커 류)이 있으면 그 안은 편집 금지.
- 이동·삭제 후 양쪽 `MEMORY.md` 인덱스 갱신.
- 처리 내역(통합/삭제/이동/인덱스) 요약 보고.

## 6. 경계 (다른 도구 영역)

- 메모리 콘텐츠 작성·저장 → 네이티브 메모리 직접 (위치/콘텐츠 분리).
- 세션 진행상황 인계 → handoff 도구.
- 프로젝트 문서 위치 → docs-manager.
- 외부 참고 자료 배치 → references-manager.
